🚀 Quick start

          
Mission Control is a local web console for your BUSY Bar — it talks to the
          device through its direct local API or the BUSY Cloud HTTP API so you can drive
          every feature from one browser page: live screen mirrors, drawing, audio, the Pomodoro-style
          BUSY timer, the file system, Wi-Fi/BLE, Matter smart-home pairing, clock, and firmware updates.

          
Direct local mode — PowerShell

          
cd busybar-web
pip install flask requests
$env:BUSY_API_BASE = "http://<device-ip>/api"
Remove-Item Env:BUSY_API_TOKEN -ErrorAction SilentlyContinue
python app.py
# then open http://127.0.0.1:8931

          
If local access mode is key, set BUSY_API_TOKEN to the device's 4–10 digit
          local key instead of removing it. You can also use http://busybar.local/api when mDNS is
          reliable; a direct LAN IP is usually faster on Windows.

          
Cloud mode — PowerShell

          
$env:BUSY_API_TOKEN = "your-BAR-scope-token"
Remove-Item Env:BUSY_API_BASE -ErrorAction SilentlyContinue
python app.py

          
Requirements

          
    
            
  • Python 3.10+ with flask and requests (pip install flask requests)
    • 
            
  • A powered-on BUSY Bar reachable through its local API or BUSY Cloud
    • 
            
  • A BAR-scope API token only for cloud mode (see API token)
    • 
          

          
How it works

          
The browser talks to a tiny Flask proxy (app.py). By default the proxy uses
          api.busy.app; set BUSY_API_BASE before launch to
          communicate directly over USB/LAN without Cloudflare or internet. Every tab calls local
          /api/… routes that map 1:1 onto the device API.

          
The green pill in the top bar shows the active API version when the selected backend is
          reachable; it turns red if the device, cloud service, or required token is unreachable.

        


        

          
🔑 API token

          
All cloud calls authenticate with an HTTP Bearer token
          (Authorization: Bearer <token>). Direct local mode needs no token when access mode is
          enabled; access mode key uses its 4–10 digit local key.

          
Where to get one

          
    
            
  1. Open the BUSY mobile/desktop app → Settings → Development → API tokens.
    2. 
            
  2. Create a BAR-scope token and copy it.
    3. 
          

          
How Mission Control uses it

          
    
            
  • No token is shipped in the source. Set BUSY_API_TOKEN before launch or use this dialog.
    • 
            
  • Click 🔑 Token in the top bar to paste a different token at runtime — it is stored
            only in the local proxy process memory, never written to disk or sent anywhere except
            api.busy.app.
    • 
            
  • The dialog shows whether a token is configured, plus a masked hint.
    • 
          

          
Security notes

          
    
            
  • Treat the token like a password — anyone holding it can fully control your device (rename it,
            flash firmware, erase files, erase Matter fabrics).
    • 
            
  • Don't commit app.py with a real token to a public repo; rotate the token if it leaks.
    • 
            
  • Because the proxy binds to 127.0.0.1, the token is only usable from your own machine.
    • 
          

        


        

          
📊 Dashboard

          
The landing tab — a live overview of the whole device, auto-refreshing.

          
Live Screens

          
    
            
  • Front (72×16, color) mirrors the native 10.0.4.20 preview: the source
            pixels are rendered as rounded LED cells inside the BUSY Bar shell.
    • 
            
  • Rear (160×80, 4-bit mono) is shown as a clean 300×150 preview with smooth 2:1
            scaling. The packed pixels are decoded low-nibble first; reversing that order swaps adjacent pixels
            and makes the OLED text unreadable.
    • 
            
  • The rear toolbar sends physical-equivalent inputs: Up, OK, Down, Back, Start / Pause,
            and a mode selector that cycles BUSY → CUSTOM → OFF → APPS → SETTINGS.
    • 
            
  • With live refresh enabled, each front/rear pair finishes before another is scheduled,
            followed by a 5 s delay. This prevents requests from stacking up when the device or cloud is slow.
    • 
            
  • ↻ Snapshot grabs one frame pair on demand. If local access becomes intermittent,
            disable live refresh, wait a few seconds, then use Snapshot. Hidden tabs pause polling automatically;
            close any additional visible Mission Control windows.
    • 
            
  • Frames are fetched via GET /busybar/screen and decoded in the browser — see
            Screen format.
    •

🖥️ Display & Audio

          
Draw on Display

          
Sends POST /display/draw with an App ID (groups your elements/assets), a
          priority (1–100; must be ≥ the on-screen app's priority — system apps are 10, an active
          work session is 90), an optional blinking LED color, and a target display (front/back).

          
Presets

          

            
PresetWhat it draws

            
Hello textscrolling text banner

            
Countdown5-minute countdown element + green LED

            
Rectanglesgradient rectangle + label

            
Stock imagebuilt-in shared/logo.png + caption

            
⚠ Alert cardpriority-95 red alert + blinking LED (preempts a running session)

          

          
Each preset also fills the Custom JSON editor so you can tweak and resend — element types:
          text, image, animation, countdown, rectangle.

          
Clear display (DELETE /display/draw) removes your app's elements — or all
          apps' elements with "Clear all".

          
Audio

          
    
            
  • Play (POST /audio/play) — plays an uploaded asset path (data.snd)
            or a stock sound (shared/…); Stop halts playback.
    • 
            
  • Volume slider (POST /audio/volume) — 0–100; "silent change" suppresses the
            confirmation beep.
    • 
            
  • Brightness (POST /display/brightness) — 0–100 or auto (ambient sensor).
    • 
          

          
Remote Input

          
Virtual button presses (POST /input?key=…): up, down, ok, back, start, busy, custom,
          off, apps, settings — same as pressing the physical keys.

        


        

          
⏱️ BUSY Timer

          
Controls the built-in focus timer. Two slots exist: busy (the standard profile) and
          custom (your secondary profile, titled "ZEN" on this device).

          
Timer snapshot

          
Shows the live state from GET /busy/snapshot: NOT_STARTED, SIMPLE
          (time left), INFINITE, or INTERVAL (current cycle, phase time left/total, pause state).

          
    
            
  • ⏸ Pause / ▶ Resume — re-PUTs the snapshot with is_paused flipped.
    • 
            
  • ■ Stop — writes a NOT_STARTED snapshot (keeps theme settings).
    • 
          

          
Start New Session

          
One click updates the slot's profile (PUT /busy/profiles/{slot}) then pushes a fresh
          snapshot — i.e. starts the timer.

          
On a fresh start (no session running) the app also sends a remote ok
          key press so the on-device CUSTOM/BUSY app foregrounds its active timer screen — the firmware only
          switches to that view on a physical OK press. On a re-start while a session is already
          running the nudge is skipped on purpose: in that state OK acts as a session control in the
          device UI and can reset/blank the timer view. The timer backend itself always (re)starts correctly
          regardless.

          
    
            
  • Simple — single countdown of N minutes.
    • 
            
  • Interval — Pomodoro: work minutes × cycles with rest minutes between, optional autostart.
    • 
            
  • Infinite — runs until stopped.
    • 
            
  • Theme — display theme (busy, on_air, …); "show work phase only"
            hides rest phases; "trigger smart home" flips the emulated switch with session state.
    • 
          

          
⚠ "Show work phase only" + INFINITE = blank screen. The flag hides everything that
          isn't a work phase — and an INFINITE timer has no work/rest phases, so the display goes dark right at
          start (and on any state transition). It only makes sense with INTERVAL profiles. The Custom quick
          controls below force it off when the profile is INFINITE.

          
Custom mode ("ZEN") quick controls

          
A dedicated card for the custom slot: start the ZEN profile with its own
          theme / show-work-phase-only / trigger-smart-home settings (loaded from the live
          profile), pause/resume/stop, a state readout, and "Open CUSTOM app on device" (sends the
          custom key so the bar's screen shows the CUSTOM app).

          
Profile editors

          
Raw JSON of each profile — edit title, timer_settings,
          busy_bar_settings and Save profile. The app refreshes
          profile_timestamp_ms automatically on save (the device uses it for sync conflict resolution).

        


        

          
🗄️ Storage & Assets

          
File browser

          
Browses the device's internal flash (/ext) via GET /storage/list. Click a
          folder to enter it, use breadcrumbs to jump back. Files can be downloaded
          (GET /storage/read) or deleted (DELETE /storage/remove).
          The usage bar comes from GET /storage/status. Create folders with mkdir
          (POST /storage/mkdir).

          
Upload

          
    
            
  • Upload to storage (POST /storage/write) — any file to any
            /ext/… path, e.g. images/animations/sounds for your apps (/ext/user_assets).
    • 
            
  • Upload as app asset (POST /assets/upload) — attaches the file to an
            App ID so display/draw and audio/play can reference it by name
            (path: "data.png").
    • 
            
  • Delete ALL assets (DELETE /assets/upload) — wipes every asset of one App ID.
    • 
            
  • Rename / move (POST /storage/rename) — old path → new path.
    • 
          

          
Paths must match ^/ext(/[a-zA-Z0-9._-]*)*$; asset/app names
          ^[a-zA-Z0-9._-]+$.

        


        

          
📡 Connectivity

          
    
            
  • Wi-Fi (GET /wifi/status) — state, SSID, BSSID, channel, RSSI, security,
            IP config. Fields only appear when connected.
    • 
            
  • Transport (GET /transport) — how the device currently talks to the cloud:
            usb or wifi.
    • 
            
  • Bluetooth LE — status (GET /ble/status: reset / initialization / disabled /
            enabled / connectable / connected), plus Enable, Disable, and
            Remove pairing (DELETE /ble/pairing) to forget the bonded phone.
    • 
            
  • HTTP API access over Wi-Fi (GET/POST /access) — controls the device's
            local HTTP API on your LAN: disabled, enabled (open), or
            key (requires a 4–10 digit PIN). This is independent of the cloud API this app uses.
    • 
            
  • MQTT backend (GET /account/backend) — broker URL, client-cert type,
            whether the server cert is ignored. Default = BUSY Cloud.
    • 
          

        


        

          
🏠 Smart Home

          
The BUSY Bar is a Matter device — it can join Apple Home, Google Home, Alexa,
          Home Assistant, etc., exposing an on/off switch that mirrors your focus session.

          
Matter Pairing

          
    
            
  1. Start commissioning (POST /smart_home/pairing) — opens a pairing window and
            returns a QR code (rendered in-page), an 11-digit manual code, and an expiry time.
    2. 
            
  2. In your smart-home app, choose "Add Matter device" and scan the QR or type the manual code before it expires.
    3. 
            
  3. Fabrics = the number of smart homes the device belongs to; "latest status" shows the
            last commissioning result (reset each power cycle).
    4. 
            
  4. Erase all fabrics (DELETE /smart_home/pairing) — removes every smart-home link.
    5. 
          

          
Emulated Switch

          
The Matter switch state (GET/POST /smart_home/switch). Toggle it manually, or let the BUSY
          timer drive it automatically ("trigger smart home" in timer settings). Startup behavior sets what
          happens on boot: off / on / toggle / last.

        


        

          
⚙️ System & Time

          
    
            
  • Device Name (GET/POST /name) — up to 20 chars, shown in apps and BLE.
    • 
            
  • Clock (GET /time) — device time with timezone. Sync from browser
            pushes your PC's local time as an ISO-8601 timestamp (POST /time/timestamp) — handy when the
            device has no NTP.
    • 
            
  • Timezone — dropdown loaded from GET /time/tzlist (all supported zones with
            offsets); Set timezone applies it (POST /time/timezone).
    • 
            
  • Firmware info (GET /status/firmware) — version, target, branch, build date,
            commit, radio (NWP) and Matter versions.
    • 
            
  • Log dump (POST /log_dump) — writes the captured device log to storage and
            returns its path; download it from the Storage tab to attach to a support ticket.
    • 
          

        


        

          
⬆️ Firmware

          
Over-the-air updates

          
    
            
  1. Check for updates (POST /update/check) — asks the cloud for a newer build;
            the status card shows the available version after a few seconds.
    2. 
            
  2. Install (POST /update/install) — downloads and applies it; the progress bar
            tracks download speed/bytes from GET /update/status. The device reboots when done.
    3. 
            
  3. Abort download (POST /update/abort_download) — cancels an in-progress download.
    4. 
            
  4. Changelog (GET /update/changelog?version=…) — release notes for any version.
    5. 
          

          
Install is blocked when the battery check fails (is_allowed: false) — plug in USB power.

          
Auto-update

          
GET/POST /update/autoupdate — enable/disable automatic installs and set the daily window
          (default 02:00–05:00).

          
Manual flash

          
POST /update uploads a firmware .tar directly — for beta/RC builds from the
          BUSY firmware repo. Warning: flashing a bad image can require recovery; only use files
          from official sources.

        


        

          
🧪 API Playground

          
A raw console that can call every documented endpoint — use it to experiment or to reach
          the few operations with no dedicated UI.

          
    
            
  1. Pick the HTTP method, then type or pick a path from the dropdown (all 48 spec paths are listed).
            You can paste full cloud URLs or /busybar/… paths — they're normalized automatically.
    2. 
            
  2. Query params — a JSON object, e.g. {"display":"0"} or {"key":"up"}.
    3. 
            
  3. JSON body — request payload for POST/PUT, e.g. {"name":"My Bar"}.
    4. 
            
  4. Send — response shows status pill (green/red), round-trip ms, and the body
            (binary bodies show a byte count).
    5. 
          

          
Examples

          
GET  /version
POST /input                    params: {"key":"busy"}
POST /display/brightness       params: {"value":"auto"}
PUT  /busy/profiles/custom     body: { …profile JSON… }
GET  /screen                   params: {"display":"1"}
POST /log_dump                 params: {"filename":"support"}

        


        

          
🖼️ Screen format (technical)

          
GET /busybar/screen?display=N is documented as image/bmp, but the body is
          actually base64-encoded raw framebuffer data, discovered by probing the device:

          

            
DisplayGeometryEncodingBytes

            
Front (0)72 × 16, colorRGB888, 3 bytes/px, row-major3456

            
Rear (1)160 × 80, monoPacked 4-bit grayscale, 2 px/byte, low nibble first, 80 bytes/row6400

          

          
The proxy base64-decodes frames and forwards raw bytes with X-Frame-Width,
          X-Frame-Height and X-Frame-Format (rgb888/gray4) headers;
          the browser expands them onto canvases. Front pixels use the native-style rounded 10×10 LED renderer;
          the rear canvas remains 160×80 internally and is displayed at 300×150. If firmware changes the format,
          the canvases show "no signal" instead of garbage.

          
rear byte 0: low nibble → x=0, high nibble → x=1
rear byte 1: low nibble → x=2, high nibble → x=3

          
Note: draw coordinates on the back display use the device's own coordinate space —
          rectangles/text positioned by x/y land where the firmware maps them onto the 160×80 buffer.

        


        

          
🩹 Troubleshooting

          

            
SymptomLikely cause → fix

            
Red "unreachable" pillDevice offline / token invalid → check the bar is powered &
            connected (USB or Wi-Fi), then set a fresh token via 🔑 Token.

            
Screens show "no signal"Frame fetch failed → device asleep or cloud hiccup; hit ↻ Snapshot.
            If persistent, firmware may have changed the frame format (see Screen format).

            
Rear text looks scrambledThe packed gray4 framebuffer is being decoded in the wrong
            nibble order → the low nibble is the left/even pixel. Use the current decoder described in Screen format.

            
Local screen requests become intermittentDisable live refresh, wait a few seconds, then
            use ↻ Snapshot. Hidden tabs pause automatically; close any additional visible Mission Control windows.

            
Draw does nothingPriority too low → a running work session is priority 90; use priority ≥ 90
            or stop the session first. Check the toast for the API error.

            
Audio silentVolume at 0, or asset path wrong → stock sounds use
            shared/…; uploaded assets must be uploaded with the same App ID you play them with.

            
Timer won't startProfile PUT failed → invalid timer_settings; use the Start New Session
            form (builds valid payloads) instead of hand-editing.

            
Screen blanks when starting/pausing in custom mode"Show work phase only" is enabled on an
            INFINITE profile → the firmware hides all non-work phases, and INFINITE has none. Turn the flag off
            (Custom quick controls do this automatically) or use an INTERVAL profile.

            
Storage 400 errorsPath pattern violation → paths must start with /ext and use
            only letters, digits, . _ -.

            
Matter pairing QR expiredCommissioning windows time out → Start commissioning again and
            scan promptly.

            
Install button blockedBattery too low → connect USB power; is_allowed becomes true.

            
App won't start locallypip install flask requests; port 8931 busy →
            PORT=9000 python app.py.

          

        


        

          
💬 FAQ

          
Does this work when the bar is on USB? Yes — the cloud API is transport-agnostic;
          the Dashboard's Network card shows whether usb or wifi is active.

          
Is my token sent to the browser? No. It lives only in the local Flask process;
          pages call 127.0.0.1 routes and the proxy attaches the token upstream.

          
Can I run this on another port / machine? PORT=8080 python app.py.
          Binding beyond localhost is not recommended (the proxy has no auth of its own).

          
What's the difference between "assets" and "storage"? Storage is the raw filesystem
          (/ext/…); assets are files namespaced to an App ID for the draw/audio APIs. An image you want
          to draw must be an asset (or a stock shared/… file).

          
Why does my draw disappear after a few seconds? Elements have timeout
          (seconds); set it higher, or re-send. Clearing happens automatically on timeout.

          
Can several apps draw at once? Yes — the firmware composites by priority and App ID;
          equal priority from a different app overrides the current one.

          
Where did the extra Python files go? The geometry probe scripts used during
          development were removed; the app folder only needs app.py and static/.
